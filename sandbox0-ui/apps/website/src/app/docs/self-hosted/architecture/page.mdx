# Architecture

Understanding sandbox0's architecture helps you make informed deployment decisions.

## High-Level Overview

Sandbox0 is designed with a clear separation between control plane and data plane:

```
                    ┌─────────────────────────────────────┐
                    │           Control Plane              │
                    │  ┌─────────────┐  ┌───────────────┐  │
                    │  │ edge-gateway│  │   scheduler   │  │
                    │  └──────┬──────┘  └───────┬───────┘  │
                    │         │                 │          │
                    │         └────────┬────────┘          │
                    │                  │                   │
                    └──────────────────┼───────────────────┘
                                       │
                    ┌──────────────────┼───────────────────┐
                    │                  ▼                   │
                    │            Data Plane                │
                    │  ┌─────────────────────────────────┐ │
                    │  │      internal-gateway           │ │
                    │  └───────────────┬─────────────────┘ │
                    │                  │                   │
                    │  ┌───────────────┼───────────────┐   │
                    │  │               │               │   │
                    │  ▼               ▼               ▼   │
                    │ ┌──────┐   ┌────────────┐   ┌─────┐  │
                    │ │manager│   │storage-proxy│   │netd │  │
                    │ └──┬───┘   └─────┬──────┘   └─────┘  │
                    │    │             │                   │
                    │    ▼             ▼                   │
                    │ ┌────────────────────────┐          │
                    │ │    Sandbox Pods        │          │
                    │ └────────────────────────┘          │
                    └─────────────────────────────────────┘
```

## Control Plane Components

### edge-gateway

The entry point for all external API requests.

**Responsibilities:**
- Request routing to appropriate data plane clusters
- Authentication and authorization
- API key and tenant management
- Token generation for internal communication

**Deployment:** Optional for single-cluster setups. Required for multi-cluster.

### scheduler

Manages template distribution across multiple data plane clusters.

**Responsibilities:**
- Template CRUD operations
- Cluster capacity tracking
- Sandbox placement decisions
- Cross-cluster template synchronization

**Deployment:** Optional for single-cluster setups. Required for multi-cluster.

## Data Plane Components

### internal-gateway

The entry point for data plane traffic.

**Responsibilities:**
- Receives requests from edge-gateway or direct clients
- Routes to appropriate manager instances
- Handles WebSocket connections for streaming

**Deployment:** Always required.

### manager

The core orchestrator for sandbox lifecycle.

**Responsibilities:**
- Maintains `SandboxTemplate` CRDs
- Manages sandbox warm pools (minIdle/maxIdle)
- Handles sandbox claim and lifecycle
- Coordinates with storage-proxy for volume mounts
- Manages network policy annotations

**Deployment:** Always required.

### storage-proxy

Manages persistent storage via JuiceFS.

**Responsibilities:**
- JuiceFS filesystem management
- Volume CRUD operations
- Snapshot creation and restoration
- gRPC file operations for sandbox pods
- S3/Object storage credential management

**Deployment:** Required for volume feature. Optional otherwise.

### netd

Node-level network policy enforcement (DaemonSet).

**Responsibilities:**
- Reads network policy from pod annotations
- Configures iptables TPROXY for transparent proxy
- Enforces L4/L7 egress rules
- Reports applied policy status

**Deployment:** Required for network policy feature. Optional otherwise.

## Data Flow

### Sandbox Creation Flow

```
1. Client Request
   │
   ▼
2. edge-gateway (auth + routing)
   │
   ▼
3. scheduler (template lookup + cluster selection)
   │
   ▼
4. internal-gateway (receives request)
   │
   ▼
5. manager (claims sandbox from pool)
   │
   ├──► storage-proxy (mount volumes if needed)
   │
   └──► netd (apply network policy if needed)
   │
   ▼
6. Sandbox Ready
```

## Storage Architecture

Sandbox0 uses JuiceFS for persistent storage:

```
┌─────────────────────────────────────────────┐
│              storage-proxy                   │
│  ┌─────────────┐    ┌─────────────────────┐ │
│  │   JuiceFS   │◄───│  PostgreSQL (meta)  │ │
│  │   Client    │    └─────────────────────┘ │
│  └──────┬──────┘                             │
│         │                                    │
│         ▼                                    │
│  ┌─────────────────────────────────────────┐│
│  │         S3 / OSS / RustFS               ││
│  │         (object storage)                ││
│  └─────────────────────────────────────────┘│
└─────────────────────────────────────────────┘
```

**Components:**
- **JuiceFS**: POSIX filesystem with metadata in PostgreSQL and data in object storage
- **PostgreSQL**: Stores filesystem metadata (inodes, directories, file chunks)
- **Object Storage**: Stores actual file data (S3, OSS, or built-in RustFS)

## Internal Authentication

Services communicate using JWT tokens signed with ed25519 keys:

```
┌─────────────────┐     JWT Token      ┌─────────────────┐
│  Control Plane  │ ─────────────────► │   Data Plane    │
│  (private key)  │                    │  (public key)   │
└─────────────────┘                    └─────────────────┘
```

**Key Pairs:**
- **Control Plane Key**: Signs tokens for control plane requests
- **Data Plane Key**: Signs tokens for data plane internal communication

Keys can be auto-generated or provided via Kubernetes secrets.

## Region and Cluster Model

### Single-Cluster

All services in one cluster, sharing one PostgreSQL and S3:

```
┌────────────────────────────────────────┐
│           Single Cluster               │
│  ┌──────────────────────────────────┐  │
│  │ internal-gateway + manager + ... │  │
│  └──────────────────────────────────┘  │
│  ┌──────────────────────────────────┐  │
│  │     PostgreSQL + S3              │  │
│  └──────────────────────────────────┘  │
└────────────────────────────────────────┘
```

### Multi-Cluster

Control plane and multiple data planes share PostgreSQL:

```
┌──────────────────────────────────────────────────────────┐
│                    Shared PostgreSQL                      │
└──────────────────────────────────────────────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│  Control Plane  │  │  Data Plane 1   │  │  Data Plane 2   │
│  edge-gateway   │  │  internal-gw    │  │  internal-gw    │
│  scheduler      │  │  manager        │  │  manager        │
└─────────────────┘  └─────────────────┘  └─────────────────┘
```

Each data plane has its own S3 bucket for isolation.
